Guia Técnico: Crie seu Framework Expert Advisor MQL5

Programar um Expert Advisor (EA) que sobreviva a volatilidade do mercado não é só questão de lógica; é lidar com latência, gerenciamento de risco e a necessidade de adaptar estratégias em tempo real. Na prática, quem tenta montar tudo em um único script acaba com código impossível de depurar, performance degradada e, pior, perdas inesperadas. Um framework bem estruturado resolve isso, oferecendo módulos reutilizáveis, separação clara de responsabilidades e pontos de extensão que permitem testar hipóteses sem refazer a base.

Por que modularizar o EA?

  • Manutenção simplificada. Cada módulo – entrada, saída, gestão de dinheiro – pode ser alterado isoladamente.
  • Teste A/B rápido. Trocar a lógica de filtragem de sinais sem tocar na camada de execução.
  • Escalabilidade. Adicionar novos indicadores ou estratégias de saída não aumenta a complexidade exponencial.

Estrutura mínima do framework

MóduloResponsabilidade
SignalGeneratorProcessa indicadores, devolve sinais (buy/sell/none).
RiskManagerCalcula lote, stop‑loss, take‑profit com base no capital disponível.
OrderExecutorAbre, modifica ou fecha ordens, tratando erros de execução.
LoggerRegistra eventos críticos para auditoria posterior.

Implementação prática – exemplo de esqueleto

 //--- SignalGenerator.mqh class SignalGenerator { public: int GetSignal() { double ma = iMA(_Symbol,0,14,0,MODE_SMA,PRICE_CLOSE,0); double prev = iMA(_Symbol,0,14,0,MODE_SMA,PRICE_CLOSE,1); if(ma>prev) return 1; // compra if(ma<prev) return -1; // venda return 0; // nada } };

O mesmo padrão se repete nos demais módulos. O ponto crítico é garantir que SignalGenerator nunca acesse a conta; ele só devolve um inteiro. Essa separação impede que um erro de cálculo altere o saldo inesperadamente – falha comum em EAs monolíticos.

Limitações e armadilhas

  • Latência de chamadas iMA pode subir se o EA for executado em múltiplos símbolos simultaneamente. A solução é pré‑calcular valores em buffers.
  • Gestão de risco baseada apenas em % de capital ignora correlação entre ativos; um portfólio diversificado requer um risk matrix mais complexo.
  • O framework não protege contra “slippage” extremo; integrar OrderCheck e monitorar OrderSendAsync é essencial em mercados de alta frequência.

FAQ rápido

  • Posso usar o mesmo framework para estratégias de scalping? Sim, mas ajuste o RiskManager para lotes menores e inclua um módulo de “SpreadFilter”.
  • E se eu precisar de múltiplos timeframes? Crie um MultiTFSignal que instancia SignalGenerator para cada timeframe e combina resultados via votação.
  • O que fazer quando o EA deixa de abrir posições? Verifique o log do Logger – normalmente um erro de OrderSend aparece antes da falha silenciosa.

Configuração inicial

O framework vem com três arquivos de configuração padrão (settings.ini, symbols.cfg e risk.cfg). Cada um controla um aspecto crítico:

  • settings.ini: parâmetros globais – horário de negociação, modo de teste, debug.
  • symbols.cfg: lista de símbolos e spreads máximos permitidos.
  • risk.cfg: risco por operação, tamanho de lote base e multiplicador de alavancagem.

Edite-os com o Notepad++ antes da primeira execução. Salve e reinicie o MetaTrader para que as alterações sejam carregadas.

Rotina recomendada para iniciantes

Dia 1 – 2: Execute o EA em modo Strategy Tester usando dados de 1‑mes de histórico. Analise o .csv gerado pelo Logger e ajuste risk.cfg se o drawdown ultrapassar 2%.

Dia 3 – 5: Mude para Visual Mode* e observe a lógica de SignalGenerator. Pause o EA sempre que um sinal parecer fora do padrão e anote a causa.

Dia 6 – 7: Ative PortfolioBalancer apenas se a conta possuir mais de três pares ativos. Monitore a variação de alocação a cada 4 horas.

Erros comuns e como evitá‑los

  • Ignorar o arquivo symbols.cfg – o EA pode tentar operar em ativos com spreads excessivos, gerando perdas inesperadas.
  • Compilar sem limpar o cache – arquivos .ex5 antigos podem permanecer e causar conflitos. Use Tools → Options → Delete All Cached Files.
  • Não ajustar o horário de negociação – operar fora das sessões de maior liquidez reduz a eficácia dos sinais.

Indicadores de progresso

Crie um mini‑dashboard no Excel com as colunas: Data, Lucro Líquido, Máximo Drawdown, % de Operações Vencedoras. Atualize‑o ao final de cada sessão de teste. Quando a taxa de vitória ultrapassar 55% e o drawdown ficar abaixo de 1,5%, considere migrar para conta real.

Quem realmente se beneficia?

Desenvolvedores que já dominam MQL5 e precisam organizar dezenas de EAs em um código reutilizável. Traders‑programadores que desejam reduzir bugs ao migrar estratégias entre contas.

Perfis incompatíveis

  • Iniciantes absolutos em programação; o framework assume familiaridade com classes, objetos e eventos.
  • Quem busca “plug‑and‑play” sem entender lógica de ordem de mercado; a abstração não substitui conhecimento de trading.
  • Usuários de plataformas que não suportam MQL5 (MetaTrader 4, cTrader, etc.).

Limitações práticas

O framework não resolve latência de execução nem falhas de conexão ao broker. Ele ainda depende de um servidor de dados de histórico estável; sem ele, otimizações perdem validade.

Mini cenários reais

Parecer editorial equilibrado

O framework entrega organização e escalabilidade para quem já fala a língua de MQL5; porém, ele adiciona camada de complexidade que pode frustrar quem busca resultados imediatos. Se seu portfólio de EAs ultrapassa cinco unidades e você sofre com manutenção, o investimento de tempo compensa. Caso contrário, continue com scripts isolados.

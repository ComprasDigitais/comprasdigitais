Guia Definitivo: Declarando Variáveis em MQL5 na Prática

Quando você abre o MetaEditor para criar um Expert Advisor, a primeira dor de cabeça costuma ser a própria sintaxe das variáveis. Não é só “escrever algo”; é garantir que o compilador entenda o tipo, o escopo e que a memória seja usada da forma mais eficiente possível. O resultado esperado é um código que compile na primeira tentativa, rode sem “segmentation fault” e ainda permita ajustes rápidos durante o back‑test. Em um cenário real – por exemplo, ao implementar um filtro de volatilidade que precisa ser recalculado a cada tick – a escolha correta da declaração pode significar a diferença entre milissegundos de latência e um EA que trava.

Tipos de Dados e Quando Usá‑los

  • int, long, short – inteiros de 32, 64 e 16 bits. Use int para contadores simples; long quando precisar de valores acima de 2 bilhões (ex.: timestamps).
  • double – ponto flutuante de 64 bits. Ideal para preços, indicadores e cálculos de risco.
  • bool – verdadeiro/falso. Evita comparações desnecessárias com 0/1.
  • string – textos curtos (nome de símbolo, logs). Cuidado: concatenação frequente gera fragmentação de memória.

Escopo: Global vs. Local

Variáveis globais vivem durante todo o ciclo de vida do EA, ocupando memória mesmo quando não são usadas. Declarações locais dentro de OnTick() são criadas e destruídas a cada chamada – mais rápido, mas potencialmente custoso se o bloco for grande. Uma prática contra‑intuitiva: mover pequenos buffers de cálculo para o escopo global pode reduzir a sobrecarga de alocação, desde que você os reinicialize explicitamente.

Inicialização e Constantes

  • Use atribuição direta na declaração: double lotSize = 0.1; – evita “garbage” antes da primeira execução.
  • Constantes com const são avaliadas em tempo de compilação, garantindo que valores críticos (ex.: spread máximo permitido) não sejam alterados por acidente.

Exemplo Prático

CódigoExplicação
input int Slippage=5; double priceOpen=0.0; bool isTradeOpen=false; const double MAX_RISK=0.02;
Inputs são configuráveis via UI; priceOpen recebe o preço do último trade; isTradeOpen controla fluxo; MAX_RISK nunca muda.

Boas Práticas de Declaração

  • Prefira int a short a menos que a economia de memória seja comprovada.
  • Evite variáveis “mágicas” espalhadas; centralize-as no início do arquivo.
  • Use #property strict para forçar verificação de tipos e evitar conversões implícitas.

FAQ Rápido

  • Posso mudar o tipo depois de compilar? Não. O código precisa ser recompilado.
  • Variáveis estáticas dentro de funções? Sim, static double sum=0; mantém valor entre chamadas – útil para médias cumulativas.
  • Qual a maior armadilha? Declarar arrays gigantes como globais sem pré‑alocação; o MetaTrader pode recusar o carregamento.

Entender esses detalhes elimina a maior parte das mensagens de erro “undeclared identifier” e permite que você foque no algoritmo em si. Se quiser aprofundar a configuração de parâmetros de entrada, dê uma olhada na documentação oficial da MQL5 – ela traz exemplos prontos que poupam tempo.

Primeiros passos após a instalação do MetaEditor

  • Abra o MetaEditor (F4) e crie um novo arquivo .mq5 via File → New.
  • Selecione Expert Advisor (EA) para ter a estrutura básica já pronta.
  • Salve o arquivo com um nome descritivo – por exemplo VariaveisDemo.mq5.

Configuração inicial da sessão de variáveis

Logo após a declaração de #property, insira um bloco dedicado às variáveis globais. Essa prática separa a lógica de negócios da configuração de estado.

//--- Variáveis globais int totalTrades = 0; // contador de trades executados double lotSize = 0.01; // tamanho padrão de lote bool isTrading = false; // flag de controle de operação string symbolTrade = "EURUSD"; // símbolo alvo

Use comentários curtos ao lado de cada declaração. Eles funcionam como documentação inline e facilitam a manutenção.

Escopo e inicialização em funções

Dentro de OnInit() ou OnTick(), evite redeclarar variáveis já existentes. Em vez disso, atribua valores conforme a necessidade do fluxo.

int OnInit() { // Inicializa apenas se ainda não houver operação em andamento if(!isTrading) { lotSize = NormalizeDouble(AccountFreeMargin()*0.02/10000,2); Print("Lot size ajustado para ",lotSize); } return(INIT_SUCCEEDED); }

Essa abordagem garante que a lógica de cálculo seja executada uma única vez, evitando sobrecarga de CPU.

Erros comuns e como evitá‑los

  • Declaração duplicada: declarar a mesma variável dentro de um bloco if cria sombra de escopo e gera valores inesperados.
  • Tipo incompatível: atribuir string a double provoca erro de compilação. Use funções de conversão como StringToDouble().
  • Inicialização tardia: deixar variáveis sem valor até o OnTick() pode gerar nan (not‑a‑number). Sempre atribua um valor padrão.

Mini‑dashboard de progresso (texto)

Inclua no log mensagens de status para monitorar o fluxo:

PrintFormat("[%s] Trades: %d | Lot: %.2f | Estado: %s", TimeToString(TimeCurrent(),TIME_DATE|TIME_SECONDS), totalTrades, lotSize, isTrading?"ATIVO":"INATIVO");

Essas linhas facilitam a auditoria sem precisar abrir o código.

FAQ rápido

  • Posso usar static para preservar valor entre ticks? Sim. Variáveis static mantêm o último valor mesmo fora do escopo da função.
  • Qual a diferença entre const e #define? const tem tipagem forte e é verificado em tempo de compilação; #define é substituição textual.
  • Quando usar enum? Para representar estados legíveis (ex.: enum TradeState {Idle, Pending, Closed}).

Para aprofundar, consulte a documentação oficial de MQL5 – ela traz exemplos prontos de uso de variáveis em contextos avançados.

Perfil ideal e limites de uso

Se você já pisa fundo em MetaTrader 5, lida com robôs e indicadores, este guia de declaração de variáveis em MQL5 fala direto ao ponto. Não é um tutorial para quem mal abriu o MetaEditor; é para quem quer dominar tipagem forte e evitar bugs silenciosos.

Quem realmente tira proveito

  • Desenvolvedores de Expert Advisors (EAs) que implementam lógica de negociação complexa.
  • Analistas quantitativos que migram de Python ou C# e precisam de controle rígido de memória.
  • Freelancers que entregam módulos reutilizáveis para clientes corporativos.

Quem provavelmente perderá tempo

  • Iniciantes que ainda não entendem a diferença entre int e double em ambientes de negociação.
  • Trader‑swing que usa apenas scripts pontuais, sem necessidade de otimização de código.
  • Usuários que dependem exclusivamente de indicadores pré‑compilados.

Limitações contextuais

O conteúdo não cobre a integração com bibliotecas externas (DLLs) nem tratamento de eventos assíncronos avançados. Também não resolve problemas de latência de rede; isso fica a cargo da arquitetura do EA.

Checklist rápido antes de fechar o código

  • Tipo de dado adequado ao valor esperado?
  • Escopo correto – global, local ou estático?
  • Inicialização explícita para evitar valores lixo.
  • Constantes definidas com const ou #define?
  • Nomeação coerente (camelCase para variáveis, UPPER_SNAKE para constantes).

Mini cenários reais

Um desenvolvedor que substituiu double por float em dezenas de cálculos de spread reduziu o consumo de RAM em 12 % e evitou overflow em backtests de alta frequência. Outro cenário: uso de const int MAGIC_NUMBER = 777001 garantiu que dois EAs não “pisassem” um ao outro ao operar contas múltiplas.
